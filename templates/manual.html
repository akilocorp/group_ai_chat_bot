ACTR User Manual

A multi-session group chat platform with configurable AI personas. Designed for research experiments where human participants interact in small groups — with or without AI bots present.

How It Works

1
Admin creates a sessionSet group size, enable bots, configure personas and their behavior on the Admin page.
2
Participants join the lobbyUsers visit the Lobby, pick a session, and click Join Matching Queue.
3
Auto-matchingOnce enough participants are queued (equal to Group Size), they are automatically matched into a small group and redirected to the chat room.
4
Group chat beginsParticipants chat in real time. If bots are enabled, they respond according to the configured mode and persona prompt.
5
Admin monitors & exportsThe Dashboard shows live activity. Full chat logs and activity history can be exported as CSV for analysis.

Bot Response Modes

Each AI persona in a session has its own Response Mode that controls when and whether it replies. Choose the mode that best matches your experimental design.

Mode 1

Fixed Delay

The bot always responds after a fixed number of seconds. Predictable and controlled.

  • Always replies
  • Exact delay you set
  • Best for controlled experiments
Mode 2

Random Delay

Adds a random 0.5–2.5 s jitter on top of the fixed delay. Feels slightly less robotic.

  • Always replies
  • Delay = fixed + random
  • Good for realism studies
Mode 3

Natural Mimicry

Delay scales with message length. Bot may skip messages based on Skip Rate.

  • May skip messages
  • Delay ∝ message length
  • Most human-like

Mode 1 — Fixed Delay (Detailed)

Mode 1 is the most transparent and reproducible mode. The bot waits exactly delay_seconds after a human message is received, then generates and sends a response. Every message gets a reply.

When to use Mode 1

  • You need consistent, reproducible bot behavior across all participants
  • You are studying the effect of response timing on human perception
  • You want to isolate the content of bot responses from timing variability
  • Pilot testing — Mode 1 is the easiest to debug and verify

How the delay works

After a human sends a message, the server waits exactly delay_seconds, then calls the OpenAI API and streams the response back. The typing animation plays while the response is being delivered. The total visible latency is delay_seconds + API response time + typing animation duration.

Tip: Set delay_seconds to 2–4 s for most experiments. Very short delays (<1 s) can feel unnatural; very long delays (>10 s) may frustrate participants.

Mode 2 — Random Delay

Identical to Mode 1 except the actual wait time is delay_seconds + uniform(0.5, 2.5) seconds. The extra jitter makes the bot feel slightly less mechanical without changing its response rate.

Use Mode 2 when you want Mode 1 behavior but with reduced suspicion that the responder is a bot.

Mode 3 — Natural Mimicry

The bot simulates a human reader: delay scales with the length of the incoming message (words × 0.25 s + random 0.5–2.0 s, capped at 8 s), and it may skip responding entirely based on skip_rate.

  • Skip Rate 0.0 — always replies (same as Mode 2 with dynamic delay)
  • Skip Rate 0.2 — skips ~20% of messages
  • Skip Rate 0.5 — skips half of all messages
Tip: Keep Skip Rate below 0.3 to avoid participants feeling ignored.

Mode 1 — All Configurable Parameters

These are all the variables you can set per-persona when using Mode 1. All are set in the Admin page under AI Personas.

ParameterDefaultWhat it controls
Delay (sec) 3 Exact seconds the bot waits before responding. This is the primary timing variable in Mode 1.
Typing Speed (CPS) 10 Characters per second for the typing animation. Higher = faster reveal. Does not affect when the response is generated, only how fast it appears on screen.
Max Tokens 150 Maximum length of the bot's response in tokens (~¾ of a word each). Lower values produce shorter, more concise replies.
Temperature 0.7 Controls response creativity. 0.0 = deterministic and repetitive; 1.0 = highly varied and creative. For research consistency, values between 0.3–0.7 are recommended.
Context Window 15 Number of recent messages the bot reads before generating a reply. Higher = more context-aware but slower and more expensive.
Idle Threshold (sec) 20 If no one speaks for this many seconds, the bot initiates a new message to restart the conversation.
Personality Prompt — The system instruction that defines the bot's persona, tone, and role. This is the most important variable for shaping what the bot says.
Bot Name — The display name shown in the chat. Participants can @mention this name to direct a message at the bot.
Session Mode 1 Controls which bots respond per message. Mode 1: all bots reply to every message. Mode 2: AI picks the single most relevant bot. Mode 3: bots only respond when @mentioned by name.
Group Duration (min) 30 Maximum duration for each group chat session in minutes. Groups are closed after this time limit.
Group Size 2 Number of human participants matched into each group. Does not include bots.
Participant Names — Optional comma-separated list of display names assigned to participants in order (e.g. Alice, Bob). If blank, UIDs are used.
Recommended Mode 1 starting config: Delay = 3 s · Typing CPS = 10 · Max Tokens = 150 · Temperature = 0.5 · Context = 15 messages

Admin Guide

Creating a session

  1. Go to Admin.
  2. Fill in Session Name, Group Size, and whether bots are enabled.
  3. Select Session Mode (Normal for most experiments).
  4. Click + Add Persona to add one or more bots.
  5. For each persona: set a name, choose Mode 1, write a personality prompt, and tune the parameters.
  6. Click Create Session. A Session ID (e.g. SES-A1B2C) is generated.
  7. Share the Lobby URL with participants. They select the session and join the queue.

Test Sandbox

Use the Admin Test Sandbox at the bottom of the Admin page to join any group as ADMIN_TESTER. Your messages are logged with that prefix so they can be filtered out of analysis.

Tip: Always test your bot configuration in the sandbox before running a real experiment session.

Dashboard & Export

Live monitoring

The Dashboard shows all active rooms, participant counts, and message volumes in real time (refreshes every 4 s). Click View Feed on any room to see its live message stream.

Exporting data

Go to the Export tab on the Dashboard. You can download:

  • Activity Log (CSV) — every event in a session: who joined, when they sent messages, when bots responded, errors, etc. One row per event.
  • Chat History (CSV) — every message in a specific room: timestamp, sender, and full message text.

Both files are ready to open in Excel, R, or Python (pandas).

Activity log columns: timestamp · event_type · session_id · room_id · actor · details
Chat history columns: timestamp · room_id · sender · message

FAQ

Can participants tell the bot is an AI?

That depends on your persona prompt and mode settings. Mode 1 with a short fixed delay is the most transparent. Mode 3 with a well-written prompt is the hardest to detect.

What happens if a participant drops out mid-session?

The remaining participants can continue chatting. The bot will still respond. The disconnected participant's messages are preserved in the log.

How many bots can I add per session?

There is no hard limit. In Session Mode 2, the AI picks the single most relevant bot per message. In Session Mode 1, all bots respond to every message. In Session Mode 3, bots only respond when @mentioned.

Where is data stored?

By default, all data is stored locally in JSON files in the db/ and activity_logs/ folders. If a MONGO_URL is set in .env, MongoDB is used instead.

How do I reset a session?

Currently there is no one-click reset. To start fresh, create a new session from the Admin page. Old sessions and their data remain on disk.
